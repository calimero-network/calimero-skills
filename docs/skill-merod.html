merod Skill

Context for running and configuring the merod Calimero node daemon

What This Skill Covers

The calimero-merod skill gives your AI assistant deep knowledge of the merod daemon so it can write correct node setup scripts, Dockerfiles, and CI health checks.

Covered Topics

  • merod init flags and defaults
  • merod run startup options
  • Config file schema and data layout
  • Health endpoint (GET /health → "alive")
  • Docker and Compose health checks
  • Multi-node port assignments

Install

npx @calimero-network/agent-skills calimero-merod

# Recommended combination:
npx @calimero-network/agent-skills calimero-merod calimero-meroctl

Two-Step Startup

# Step 1: Init (ONCE on a fresh home)
merod --node node1 init --server-port 2428 --swarm-port 2528

# Step 2: Run (every time after)
merod --node node1 run

# Custom home directory
merod --home ./data --node node1 init --server-port 2428 --swarm-port 2528
merod --home ./data --node node1 run

# Multiple nodes on one machine (different ports)
merod --home ./data --node node1 init --server-port 2428 --swarm-port 2528
merod --home ./data --node node2 init --server-port 2429 --swarm-port 2529

Port Roles

--server-port (2428)
HTTP/WS API — connects to meroctl, browser frontends, Python client, JS clients. Must be reachable by all API consumers.
--swarm-port (2528)
P2P swarm — connects to other merod nodes for state sync and namespace invite protocol. Must be reachable between nodes in multi-node setups.

Health Check

GET http://localhost:2428/health
→ 200 OK: { "status": "alive" }

# Wait for node ready (shell)
until curl -sf http://localhost:2428/health > /dev/null; do
  echo "waiting..."; sleep 1
done

# Docker health check
HEALTHCHECK --interval=5s --timeout=3s --retries=10 \
  CMD curl -sf http://localhost:2428/health || exit 1

Note: The status value is "alive", not "healthy". Code checking for "healthy" will incorrectly report a live node as unhealthy.

Key Rules

Always init before run
Running merod run on an uninitialised home fails immediately — no config, no keypair, no storage.
Never re-init production
Re-running merod init generates a new keypair and overwrites config, destroying the node identity and breaking all existing context memberships.
Ports set at init time
Port numbers are written into config.toml during init. You cannot override them with merod run flags. Edit config.toml if you need to change them.
